# 接收代码审查

## 概述

代码审查需要技术评估，而不是情绪表演。

**核心原则：** 先验证，再实现；先提问，再假设。技术正确性高于社交舒适感。

## 响应模式

```
收到代码审查反馈时：

1. 读取：先完整看完反馈，不要立刻反应
2. 理解：用自己的话复述需求（不清楚就直接问）
3. 验证：对照代码库现实情况检查
4. 评估：对这个代码库来说，技术上是否站得住？
5. 回应：技术性确认，或者给出有理有据的反驳
6. 实施：一次只做一项，每项都单独测试
```

## 禁止的回应

**NEVER:**
- “你完全正确！”（明确违反 `CLAUDE.md`）
- “说得好！” / “反馈很棒！”（表演式回应）
- “我现在就实现” （还没验证之前）

**应该这样：**
- 复述技术要求
- 提出澄清问题
- 如果有问题，用技术理由反驳
- 直接开始做事（行动 > 口头表态）

## 处理不清晰的反馈

```
如果任何一项不清楚：
  立刻停止 - 先不要实现任何东西
  先把不清楚的地方问明白

原因：这些项可能彼此相关。理解不完整 = 实现就会错。
```

**Example:**
```
你的人工协作者：“修复 1-6”
你理解了 1、2、3、6，但 4、5 不清楚。

❌ 错误：先实现 1、2、3、6，之后再问 4、5
✅ 正确：“我理解 1、2、3、6。开始前需要先确认 4 和 5。”
```

## 按来源处理

### 来自你的人工协作者
- **可信** - 理解后就实现
- **范围不清** 还是要问
- **不要表演式附和**
- **直接行动** 或给出技术性确认

### 来自外部审查者
```
实现之前：
  1. 检查：对这个代码库来说技术上正确吗？
  2. 检查：会破坏现有功能吗？
  3. 检查：当前实现为什么会这么写？
  4. 检查：在所有平台/版本上都能工作吗？
  5. 检查：审查者是否理解完整上下文？

如果建议看起来不对：
  用技术理由反驳

如果没法轻易验证：
  直接说明：“没有 [X] 我没法验证。要我 [调查/询问/继续] 吗？”

如果和你的人工协作者之前的决定冲突：
  先停下来和你的人工协作者讨论
```

**你的人工协作者的规则：** “外部反馈要保持怀疑，但要认真检查。”

## “专业化”功能的 YAGNI 检查

```
如果审查者建议“正确地实现”：
  先在代码库里 grep 实际使用情况

  如果没用到：“这个接口没有被调用。要不要删掉（YAGNI）？”
  如果有使用：那就按正确方式实现
```

**你的人工协作者的规则：** “你和审查者都向我负责。如果不需要这个功能，就不要加。”

## 实施顺序

```
对于多项反馈：
  1. 先把不清楚的地方问明白
  2. 然后按这个顺序实现：
     - 阻塞问题（会破坏功能、安全问题）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 每项修复都单独测试
  4. 确认没有回归
```

## 何时反驳

出现以下情况时要反驳：
- 建议会破坏现有功能
- 审查者缺少完整上下文
- 违反 YAGNI（功能没用到）
- 对这个技术栈来说是错的
- 存在遗留/兼容性原因
- 与你的人工协作者的架构决定冲突

**How to push back:**
- 用技术理由，不要带防御情绪
- 提出具体问题
- 引用现有测试或代码
- 如果是架构问题，带上你的人工协作者一起讨论

**如果你不方便直接说反驳话，可以用这句信号：** “事情有点不对劲。”

## Acknowledging Correct Feedback

当反馈确实正确时：
```
✅ “已修复。[简要说明改了什么]”
✅ “发现得好 - [具体问题]。已在 [位置] 修复。”
✅ [直接修复并在代码里体现]

❌ “你完全正确！”
❌ “说得好！”
❌ “感谢你指出来！”
❌ 任何形式的感谢
❌ 任何感激表达
```

**为什么不能说谢谢：** 行动胜于语言。直接修复就行，代码本身会说明你已经接收了反馈。

**如果你发现自己快要写出“谢谢”：** 直接删掉，改成说明修复内容。

## Gracefully Correcting Your Pushback

如果你反驳后发现自己错了：
```
✅ “你是对的 - 我检查了 [X]，它确实会 [Y]。现在开始修。”
✅ “我核实了一下，你说得对。我的初始理解错在 [原因]。正在修复。”

❌ 长篇道歉
❌ 为之前的反驳辩解
❌ 过度解释
```

要事实性地说明修正，然后继续推进。

## Common Mistakes

| 错误 | 修复方式 |
|---------|-----|
| 表演式附和 | 直接说需求，或者直接做 |
| 盲目实现 | 先对照代码库验证 |
| 批量实现不测试 | 一次一个，每个都测试 |
| 先入为主认为审查者一定对 | 先看会不会破坏现有逻辑 |
| 回避反驳 | 技术正确性 > 舒适感 |
| 部分实现 | 先把所有条目问清楚 |
| 无法验证还继续 | 说明限制，询问方向 |

## Real Examples

**表演式附和（坏例子）：**
```
审查者：“删掉遗留代码”
❌ “你完全正确！我这就删……”
```

**技术验证（好例子）：**
```
审查者：“删掉遗留代码”
✅ “我看一下……当前构建目标是 10.15+，这个 API 需要 13+。为了向后兼容需要保留旧实现。现在实现的 bundle ID 也不对 - 是修它，还是直接放弃对 13 之前的支持？”
```

**YAGNI（好例子）：**
```
审查者：“把指标追踪完整实现出来，要数据库、日期筛选、CSV 导出”
✅ “我 grep 了代码库，没看到任何地方调用这个接口。要不要删掉（YAGNI）？还是我漏看了使用点？”
```

**不清楚条目（好例子）：**
```
你的人工协作者：“修复 1-6”
你理解了 1、2、3、6，4、5 不清楚。
✅ “我理解 1、2、3、6。实现前需要先确认 4 和 5。”
```

## GitHub 线程回复

在 GitHub 上回复行内审查评论时，要在评论线程里回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），不要作为顶层 PR 评论回复。

## 最终原则

**外部反馈 = 需要评估的建议，不是必须照做的命令。**

先验证。再质疑。然后实现。

不要表演式附和。始终保持技术严谨。
