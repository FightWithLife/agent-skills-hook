# 架构图工作流

## 作用

用户要求先写架构文档或修改架构文档时，就先写一份架构图文档，再进入实现。

图怎么画、怎么命名、怎么渲染、怎么重渲染，统一遵循 [diagram-workflow](../diagram-workflow/SKILL.md)。

架构文档默认落到 `docs/plans/YY-MM-DD_name/architecture/<feature-name>.puml` 和 `docs/plans/YY-MM-DD_name/architecture/<feature-name>.md`。

默认优先用于嵌入式 C / 纯 C / Rust / HTML 的改动，尤其是 BSP、驱动、协议、板级初始化和静态页面这类需要看边界的工作。

## 联动顺序

1. 先用这个 skill 把架构文档写出来或改完。
2. 再交给 `diagram-workflow`，把同一份架构文档细化成图并渲染。
3. 架构文档和图都确认后，再进入 `writing-plans`，把已确认的设计拆成执行计划。

## 基本流程

1. 先判断这次改动属于哪一类：
   - 现有代码修改
   - 从零搭建新系统
   - 新增独立功能模块
2. 先写架构文档里的图和说明。
3. 按 `diagram-workflow` 生成和刷新图。
4. 检查架构文档是否清楚、是否过挤、是否容易比较。
5. 如果看不清，先简化图，再重新渲染。

## 图怎么选

- 架构规划：画组件图、边界图，先看模块关系和职责边界
- 接入点：画编号接入点图，先看改动从哪里接入
- 功能流程：画流程图或活动图，先看主路径和关键分支
- 修改对比：画 before / after 对照图，先看前后变化

如果不确定，优先选“架构规划”，再补接入点和流程图。

## 最低验收

在实现开始前，文档至少要讲清楚：

- 这次改什么
- 接到现有代码的哪里
- 主流程怎么走
- 修改前后差了什么
