# 项目架构文档初始化

## 定位

这不是代码说明文档，而是**代码导航文档**。目标是让 AI（和人）能快速定位到正确代码位置，避免在仓库里大海捞针。

- 每个模块独立成文，模块文档要讲清楚：**模块在哪、干什么、怎么走**
- 最关键的是**仓库目录映射**：明确写出模块对应的实际路径，防止 AI 找错地方

## 与实现架构的区别

- **本 skill**：初始化项目的整体架构文档系统，产出模块级导航文档。与具体代码修改无关，属于项目级长期文档。
- **`implementation-architecture-workflow`**：针对某次代码改动，编写改动范围内的实现架构文档。属于改动的临时设计文档。

## 触发条件

- 新项目启动，需要建立架构文档体系
- 现有项目缺少架构文档，需要补建
- 用户明确要求"写架构文档"且不是针对某次代码改动

## 基本流程

1. 扫描仓库目录结构，识别模块边界（见下方"模块边界识别"）
2. 写架构总览 `01_架构总览.md`，按"架构总览模板"编写
3. 为每个模块写独立导航文档，放入 `modules/` 目录，按"模块文档模板"编写
4. 需要图示时，用 `plantuml-ascii` 绘制，图源放入 `diagrams/`
5. 创建或更新仓库根目录的 `AGENTS.md`，写入架构文档导航规则（见下方"AGENTS.md 规则注入"）
6. 全部完成后做验收检查（见"验收标准"）

## 模块边界识别

按以下优先级判定模块边界：

1. **目录即模块**：如果仓库已经按目录划分了子系统/组件，直接以目录为模块单位
2. **入口文件回溯**：找 `main()`、`app init`、注册函数等入口，反向追踪调用链来聚合文件
3. **独立构建目标**：如果某目录有独立的 `CMakeLists.txt`、`Makefile`、`Cargo.toml` 等，就是一个模块
4. **平铺文件兜底**：如果几十个文件平铺在一个目录里，按功能前缀分组（如 `uart_*.c`、`spi_*.c`），每组为一个模块
5. **嵌套处理**：模块内若有明显的子模块边界（如 `module/sub_a/`、`module/sub_b/`），子模块独立成文，编号用两级序号（如 `02-01_子模块名.md`）

不确定边界时，宁可多拆不要合并——小模块文档比大杂烩有用。

## 文档目录规划

```
docs/architecture/
├── 01_架构总览.md            # 项目全景：模块列表、模块关系总图、技术栈
├── modules/                  # 各模块独立导航文档
│   ├── 02_<模块名>.md        # 按序号命名，子模块用 02-01_<子模块名>.md
│   ├── 03_<模块名>.md
│   └── ...
├── diagrams/                 # PlantUML 源文件
│   └── *.puml
└── 架构决策/                 # 架构决策记录
    └── YY-MM-DD_决策主题.md
```

## AGENTS.md 规则注入

所有架构文档生成后，在仓库根目录创建或更新 `AGENTS.md`，写入以下内容（按实际模块名和路径调整）：

```markdown
<!-- AGENTS.md — 由 project-architecture-init 自动生成 -->

## 项目架构文档

本仓库已建立项目级架构文档，位于 `docs/architecture/`。

### 读写规则

- 在修改代码之前，先按需阅读相关模块的架构文档，理解模块职责和边界后再动手
- 架构变更时（新增/删除模块、文件迁移、职责变化），同步更新 `docs/architecture/` 下对应的文档和本规则
- 如果架构文档中标注了"仓库目录映射"，以映射路径为准，不要靠推测找文件

### 文档索引

| 文档 | 路径 | 说明 |
|------|------|------|
| 架构总览 | `docs/architecture/01_架构总览.md` | 模块全景图、模块列表、顶层目录结构 |
| 模块A | `docs/architecture/modules/02_模块A.md` | ... |
| 模块B | `docs/architecture/modules/03_模块B.md` | ... |
```

如果 `AGENTS.md` 已存在，检查是否已有架构文档相关段落，有则更新内容，无则追加在合适位置。

`01_架构总览.md` 按以下结构编写：

```markdown
# 项目架构总览

## 项目简介
- 项目名称：<名称>
- 技术栈：<语言/框架/平台>
- 一句话定位：<这个项目是干什么的>

## 模块全景图
<!-- 用 plantuml-ascii 绘制整体模块关系图，嵌入 ASCII -->

## 模块列表

| 序号 | 模块名 | 文档 | 一句话职责 |
|------|--------|------|-----------|
| 02   | <模块A> | [02_<模块A>.md](modules/02_<模块A>.md) | <职责> |
| 03   | <模块B> | [03_<模块B>.md](modules/03_<模块B>.md) | <职责> |
| ...  | ...    | ...  | ...       |

## 仓库顶层目录结构
<!-- 列出仓库根目录结构，标注每个目录与哪个模块相关 -->

- `src/` — 主源码，包含模块 <A>、<B>
- `include/` — 公共头文件
- `tests/` — 测试代码
- `config/` — 配置文件
- ...
```

## 模块文档模板

每个 `modules/` 下的模块文档按以下结构编写：

```markdown
# <模块名>

## 概述
<一句话说清这个模块是干什么的，核心职责是什么>

## 仓库目录映射
<!-- 这是最重要的部分：告诉 AI 代码在哪 -->
- 主目录：`src/<实际路径>/`
- 核心文件：
  - `src/<路径>/xxx.c` — <文件职责>
  - `src/<路径>/xxx.h` — <头文件职责>
  - `src/<路径>/yyy.c` — <文件职责>
- 测试目录：`tests/<路径>/`
- 相关配置：`config/<路径>/`
- 依赖的其他模块：`<模块B>`（源码在 `src/<路径B>/`）

## 架构图
<!-- 用 plantuml-ascii 绘制，嵌入 ASCII 图 -->

## 关键流程
<!-- 核心业务/数据流程，一图一事，嵌入 ASCII 流程图 -->

## 主要数据结构
<!-- 模块内关键结构体、类、接口，每个附简要说明 -->

### <结构体/类名>
- 定义位置：`src/<路径>/xxx.h:行号`
- 用途：<说明>
- 关键字段：
  - `field_a` — <含义>
  - `field_b` — <含义>
```

## 图怎么画

使用 `plantuml-ascii` 绘制 ASCII 架构图。`plantuml.jar` 在 `plantuml-ascii` skill 的同级目录下。

优先画：
- 系统边界图（组件图）：看清系统与外部的关系
- 模块关系图（组件图）：看清内部模块划分与依赖
- 核心数据流图（活动图/时序图）：看清主要数据处理路径

画图步骤：
1. 编写 `.puml` 源文件，放入 `diagrams/` 目录
2. 从 `plantuml-ascii` skill 目录找到 `plantuml.jar`，执行 `java -jar <skill目录>/plantuml.jar -utxt <file>.puml`
3. 生成的 `.utxt` 文件内容嵌入对应的架构文档

## 验收标准

初始化完成后，逐项检查：

| 检查项 | 标准 |
|--------|------|
| 模块覆盖 | 所有可识别模块都有对应文档，无遗漏 |
| 路径可定位 | 每个模块文档的"仓库目录映射"指向真实存在的目录/文件 |
| 图完整 | 架构总览有模块关系图，每个模块文档至少有一张图 |
| 交叉引用正确 | 模块间的依赖关系在文档中有标注且路径正确 |
| 一致命名 | 模块名、文档名、图中标签三者一致 |
| AGENTS.md | 仓库根目录存在 `AGENTS.md`，且包含架构文档索引和读写规则 |

## 文档维护

导航文档不是一次性的。以下情况应更新：

- 新增/删除模块时，更新 `01_架构总览.md` 模块列表，新增/删除对应模块文档，同步更新 `AGENTS.md` 文档索引
- 模块文件迁移时，更新对应模块文档的"仓库目录映射"
- 模块职责发生重大变化时，更新概述、架构图和关键流程
- 新增关键数据结构时，在对应模块文档的"主要数据结构"中补充

维护时优先保证"仓库目录映射"的准确性——路径错了，整个文档就失去导航价值。

## 完成后

架构文档初始化完成后，不必自动进入实现工作流。仅在用户后续发起具体代码改动时，才按需要触发 `implementation-architecture-workflow`。
