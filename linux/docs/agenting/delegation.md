# 委派策略

本文档说明 Claude 和 Codex 在本仓库中的协作边界。

## 总原则

- 小任务直接做，大任务先拆分
- 先判断是否需要架构确认，再判断是否需要验证分工
- 不因为“看起来中等”就自动升级

## 适合直接处理

- 单文件文本修改
- 注释、命名和说明清理
- 不改构建图、不碰硬件边界的局部逻辑修复

## 适合升级

- 多文件改动且存在共享边界
- 构建失败、链接失败或生成文件不一致
- 触及 ISR、共享状态、`volatile`、寄存器、DMA、时钟或板级初始化
- 需要明确分工的验证、评审或风险复核

## 分工建议

- 架构不清时，先看 `docs/agenting/architecture.md`
- 验证入口不清时，先看 `docs/agenting/verification.md`
- 需要多代理协调时，再看本文件

## 协作规则

- 主代理负责拆解、委派、整合和最终结论
- 子代理只执行已分配的范围，不自行扩大边界
- 如果范围变化，先重新定义任务，再继续执行
- 证据优先于推测，结论必须能回溯到文件或命令

## 典型路由

- 只读调查：优先收集事实，再决定是否需要实现
- 构建修复：优先解决工具链和配置问题
- 硬件风险：优先做风险复核，再进入实现
- 最终回归：必须基于实际验证证据
