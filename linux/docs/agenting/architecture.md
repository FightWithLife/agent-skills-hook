# 架构边界

本文档说明本仓库相关提示词和配置应如何划分架构责任。

## 目标

- 保持入口文件短小明确
- 将长规则沉到深文档，避免上下文膨胀
- 让 Claude 和 Codex 对同一套边界有一致理解

## 推荐分层

- `AGENTS.md`：共享入口，只放导航、硬边界和最小约束
- `claude/CLAUDE.md`：Claude 运行时增量，保留 Claude 专属行为
- `codex/AGENTS.md`：Codex 运行时增量，保留 Codex 专属行为
- `docs/agenting/*.md`：放长规则、检查清单、委派和验证细则

## 适用范围

- 嵌入式 C 工作流
- `Make` / `CMake` 构建与修复
- 板级支持、启动顺序、外设和时钟相关变更
- 与硬件行为相关的风险控制

## 架构原则

- 共享入口只描述事实和边界，不写操作手册
- 运行时差异只放到对应入口文件，不在共享入口里重复
- 深规则按主题拆分，不要把验证、委派和架构全塞进一个文件
- 任何会影响板级行为、链接行为或初始化顺序的改动，都要显式说明边界

## 何时升级

- 模块边界不清
- BSP、HAL、驱动、协议和应用层的职责需要重划
- 启动顺序、板级初始化或时序需要变化
- 需要跨文件协调、跨层验证或硬件风险隔离

## 不建议

- 在入口文件中堆积大段背景说明
- 把个人偏好、临时流程或会话输出格式写成仓库共享规则
- 用抽象口号替代真实的边界定义
