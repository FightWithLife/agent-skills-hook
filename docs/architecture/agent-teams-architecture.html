Two-Agent-Team Architecture

Documentation implementation for OpenCode AGENTS workflows

Team A - Development Team

orchestrator Primary planner and dispatcher for development work. Model: GPT-5.2
dev Implements features and fixes with verifiable outputs. Model: GPT-5.3 Codex
qa Runs acceptance and regression checks. Model: GPT-5.3 Codex
review Performs structured risk-focused code review. Model: GPT-5.2 Codex
debug Auto-callable quick localization pass for hard issues. Model: GPT-5.3 Codex
Flow: User request -> orchestrator -> (dev, qa, review) -> integrated result. If a hard issue appears, orchestrator can auto-dispatch one quick debug pass.

Team B - Hard-Problem Team (Manual Trigger)

triage Manual-only root-cause lead. Runs hypothesis loops. Model: GPT-5.2
dev Verifies candidate fixes against narrowed focus points. Model: GPT-5.3 Codex
debug (shared) Shared from Team A for focused localization loops. Model: GPT-5.3 Codex
Flow: Manual @triage -> uses shared debug verification loops -> focused hypotheses -> dev fix validation -> root-cause report.

Implementation Method

  • Keep one shared role vocabulary across platforms.
  • Write OpenCode AGENTS instruction blocks with matching role contracts.
  • Use fixed task protocol fields for dispatch and result collection.
  • Enforce manual-only triage invocation policy.
  • Preserve evidence-first output in debug, qa, and review.

Escalation Policy

  • `orchestrator` runs one quick `debug` pass on suspected hard issues.
  • If unresolved, suggest user to manually invoke `triage`.
  • No automatic promotion from `orchestrator` to `triage`.
  • `triage` is manual-only; `debug` is not manual-only.
This architecture is instruction-driven. Runtime orchestration is controlled by OpenCode AGENTS behavior and user invocation policy.
