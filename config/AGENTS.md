# agent-skills-hook 仓库总则

本仓库用于维护 Claude Code、Codex 和 OpenCode 的协作提示词、配置和相关文档。

## 使用原则

- 先读深文档：`docs/agenting/architecture.md`、`docs/agenting/verification.md`、`docs/agenting/delegation.md`、`docs/agenting/runtime-style.md`
- 入口文件只保留导航、边界和最小约束，不重复长规则
- Claude/Codex 各自的运行时入口文件只保留对应增量，不把共享规则复制回入口
- 默认面向嵌入式 C、`Make` / `CMake`、构建修复和硬件风险控制
- 修改前先确认范围，尽量保持最小 diff
- 不改无关文件，不引入个人路径，不把运行时偏好写进共享入口
- 默认先交付最小闭环可复核结果，再做增强或重构
- 信息不全时先做合理假设推进；但不要假定用户已经完全厘清需求，遇到目标、范围或取舍不明确时，先向用户确认再执行

## 共享边界

- 适合本仓库的改动：提示词、代理路由、构建与验证说明、轻量配置模板
- 不适合在入口文件里展开的内容：具体检查清单、长流程、示例命令、深层协作规则
- 需要结构性调整时，优先参考 `docs/agenting/architecture.md`
- 需要验证或回归时，优先参考 `docs/agenting/verification.md`
- 需要委派或多代理协调时，优先参考 `docs/agenting/delegation.md`
- 需要统一运行时处理方式或沟通风格时，优先参考 `docs/agenting/runtime-style.md`

## 构建与验证

- 构建规则以仓库内真实工具链为准，优先使用现有 `make`、`cmake --build`、脚本或板级命令
- 涉及 `Make`、`CMake`、链接、包含路径、宏或生成代码时，必须说明验证依据
- 涉及中断、共享状态、`volatile`、寄存器、DMA、时钟或板级初始化时，必须补充风险检查
- 如果缺少验证入口，先补齐最小可复核证据，再谈结论
- 结论必须附带验证结果和未覆盖风险，不做空泛通过判断

## 委派准则

- 单文件、低风险、无硬件面影响的改动，优先就地处理
- 多文件但边界清楚的改动，先对齐架构，再分阶段实施
- 触及构建失败、链接失败、协议边界、硬件风险或跨模块影响时，升级到对应深文档流程
- 不在入口文件里写复杂协作协议，复杂规则下沉到 `docs/agenting/delegation.md`
- 运行时风格、并行协作节奏和子代理回收规则属于运行时入口，不写进共享入口

## 产出要求

- 中文说明优先，术语保持一致
- 说明要能被 Claude 和 Codex 共同引用
- 如果任务是 review，结论部分使用中文，并保留可追溯证据
- 超过一屏的细则默认下沉到 `docs/agenting/*.md` 或对应运行时子目录
