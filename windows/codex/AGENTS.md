# Codex 运行时入口（agent-skills-hook）

这些说明会被 Codex CLI 全局加载。本文件为 Windows 版本，自包含所有规则。

## SessionStart（每个会话一次）
- 在每个新会话的首条回复中输出一个简短区块：
  - `SessionStart` 标题
  - 当前生效的指令层（全局 `~/.codex/AGENTS.md`、仓库内 `AGENTS.md`，如果存在）
  - 技能来源（`~/.agents/skills`、`./.agents/skills`，通过相对路径引用 `../../agents/skills`）
  - Execpolicy 规则路径（`~/.codex/rules/*.rules`）

## 技能（每次用户请求）
- 在任何实质性工作开始前，先用 Codex 原生技能机制判断是否有可匹配的技能。
- 使用 Codex 原生技能语义：
  - 显式调用使用 `$skill-name`；
  - 当请求明显匹配某个技能的 `description` 时，可以隐式调用；
  - 只有在真正选中的技能上才加载完整的 `SKILL.md` 内容。
- 不要要求或模拟 `Skill(...)`、`skills(...)` 这类非原生函数式调用。
- 优先读取仓库和用户层面的技能元数据；仅在需要时再读取所选技能的文件。
- 如果当前运行时没有原生技能选择界面或原生技能提及能力，就用普通文件/命令工具手动读取匹配的 `SKILL.md`，并安静执行。
- 不要为了凑技能流程而输出套话、伪造工具调用或执行空操作命令。

## 工具安全
- 遵守 execpolicy 规则。除非用户明确要求且这样做是安全的，否则不要绕过安全检查。

## 复核输出语言
- 当用户要求 `review` 时，最终回复使用中文，同时保持规定的 review 结构和格式规则不变。

## 架构原则
- 保持入口文件短小明确，避免上下文膨胀。
- 共享入口只描述事实和边界，不写操作手册。
- 运行时差异只放到对应入口文件，不在共享入口里重复。
- 深规则按主题拆分，不要把验证、委派和架构全塞进一个文件。
- 任何会影响板级行为、链接行为或初始化顺序的改动，都要显式说明边界。

### 适用范围
- 嵌入式 C 工作流
- `Make` / `CMake` 构建与修复
- 板级支持、启动顺序、外设和时钟相关变更
- 与硬件行为相关的风险控制

### 何时升级架构审查
- 模块边界不清
- BSP、HAL、驱动、协议和应用层的职责需要重划
- 启动顺序、板级初始化或时序需要变化
- 需要跨文件协调、跨层验证或硬件风险隔离

## 嵌入式 C 协作模式
- 面向以 C 为主、使用 `Make` 或 `CMake` 的嵌入式固件工作进行优化。
- 默认保持轻量；如果直接回答或单文件修复就够了，就不要升级到额外子代理。
- 当任务跨越架构、构建、硬件、并发或多文件风险边界时，再升级到更强的执行路径。
- 常见路由按职责区分：`explorer` 做只读调查，`architect` 做结构边界，`build-resolver` 处理构建问题，`firmware-reviewer` 和 `hardware-impact` 复核固件/硬件风险，`reviewer` 做最终通用回归检查。

## Codex 内置子代理

### 核心子代理（常用）
| 名称 | 调用方式 | 职责 | 适用场景 |
|------|---------|------|---------|
| `explorer` | `--agent explorer` | 只读调查，负责定位文件、配置、调用路径与上下文 | 需要快速定位、错误排查、代码理解 |
| `worker` | `--agent worker` | 受控实现，负责小步改动、最小差异、可验证输出 | 实现功能、修复 bug、代码变更 |
| `reviewer` | `--agent reviewer` | 最终回归评审，负责正确性、可维护性、风险与证据 | 代码审查、回归检查、质量验证 |
| `planner` | `--agent planner` | 多步骤规划，负责拆分阶段、验收标准、风险与回滚点 | 复杂任务规划、架构设计、项目拆解 |
| `build-resolver` | `--agent build-resolver` | 构建/工具链修复，负责编译、链接、依赖、生成步骤 | 构建失败、依赖问题、工具链修复 |
| `architect` | `--agent architect` | 结构设计与评审，负责模块边界、职责划分、演进路径 | 架构设计、重构规划、结构评审 |

### 可选子代理（按需启用）
| 名称 | 调用方式 | 职责 | 适用场景 |
|------|---------|------|---------|
| `hardware-impact` | `--agent hardware-impact` | 硬件侧影响评估与回归面梳理 | 涉及硬件、驱动、寄存器的改动 |
| `firmware-reviewer` | `--agent firmware-reviewer` | 低层并发、内存、时序等风险审查 | 固件开发、实时性、并发控制 |
| `monitor` | `--agent monitor` | 状态观察，负责长耗时构建、测试、日志监控与简明汇报 | 长时间任务、构建监控、日志分析 |

### 调用示例

```bash
codex --agent explorer "定位登录失败的根本原因"
codex --agent worker "修复用户管理模块的内存泄漏"
codex --agent build-resolver "修复链接错误：undefined reference to 'init_config'"
codex --agent architect "重构认证模块，支持多因子验证"
codex --agent reviewer "审查最近的改动，检查是否引入回归"
```

## 委派策略

### 总原则
- 小任务直接做，大任务先拆分。
- 先判断是否需要架构确认，再判断是否需要验证分工。
- 不因为"看起来中等"就自动升级。

### 适合直接处理
- 单文件文本修改
- 注释、命名和说明清理
- 不改构建图、不碰硬件边界的局部逻辑修复

### 适合升级
- 多文件改动且存在共享边界
- 构建失败、链接失败或生成文件不一致
- 触及 ISR、共享状态、`volatile`、寄存器、DMA、时钟或板级初始化
- 需要明确分工的验证、评审或风险复核

### 自动委派规则
| 场景关键词 | 自动选择子代理 |
|-----------|---------------|
| 定位、排查、错误原因、调用路径 | `explorer` |
| 实现、修复、代码变更 | `worker` |
| 审查、回归、验证、评审 | `reviewer` |
| 规划、拆分、阶段、验收标准 | `planner` |
| 编译失败、链接错误、构建问题 | `build-resolver` |
| 架构、重构、模块边界 | `architect` |
| 硬件、寄存器、驱动 | `hardware-impact`（可选）|
| 固件、并发、时序、中断 | `firmware-reviewer`（可选）|

### 协作规则
- 主代理负责拆解、委派、整合和最终结论。
- 子代理只执行已分配的范围，不自行扩大边界。
- 如果范围变化，先重新定义任务，再继续执行。
- 证据优先于推测，结论必须能回溯到文件或命令。

## 多代理原则
- 任务可以拆成独立子任务且确有收益时，优先按职责分派。
- 单文件、低风险修复保持单代理。
- 子代理只负责执行已分配范围，不递归委派，也不抢占其他子代理的作用域。
- 能并行就并行，但仅在任务可拆且不会造成文件或环境冲突时启用。
- 临时子代理在输出整合后应尽快关闭。

### 标准并行开发流程
适用于复杂多文件、跨架构、跨构建的任务。

#### 1. 0-60秒产出子任务
- 收到需求后，最多60秒内产出可分配的子任务清单。
- 每个子任务标注：边界、预期产出、验证入口、预估时间。

#### 2. 并行启动（Codex 内置子代理）
根据任务复杂度选择合适的子代理组合：
- **轻量调试/定位**：`explorer` + `worker`
- **复杂实现**：`planner` + `worker` + `reviewer`
- **构建修复**：`build-resolver` + `explorer`
- **架构重构**：`architect` + `planner` + `worker` + `reviewer`
- **嵌入式 C 全流程**：`explorer` → `planner` → `architect` → `worker` → `build-resolver` → `hardware-impact` → `firmware-reviewer` → `reviewer`

#### 3. 子 Agent 开工模板
每次开工输出一次：
```
[Agent-<名称> 开工]
目标：<一句话任务边界>
负责：文件/目录范围
产出：预期交付物
验证：验证入口（测试/命令/检查点）
预计：时间预估
```

#### 4. 主控集成
主控负责：边界决策、任务调度、冲突仲裁、进度跟踪、最终验收、输出汇总。

#### 5. 交付模板
```
结果：<一句话总体状态>
Agent进度：explorer/worker/... 完成项
改动：文件/命令/配置变化
验证：执行项与结果
风控：<回归风险与影响面>
下一步：最多3条
```

#### 子 Agent 生命周期
- 关闭动作：完成后立即上报主控，等待回收指令
- 心跳：长任务每30秒输出一次进度（一句话）
- 空闲回收：无产出超过60秒，主控可回收
- 超时回收：超过预估时间2倍，主控强制回收
- 失败处理：失败立即上报，主控决定重试或降级

#### 冲突控制
- 禁止多 Agent 同时写同一文件。
- 文件锁定：主控负责分配文件边界，子 Agent 只在分配范围内操作。
- 发现冲突时：立即停止，上报主控仲裁。

## 默认执行节奏
- 默认以可落地结果为优先，不停留在空方案或空建议。
- 先拆成 3-5 个任务块；可并行的探索、实现和验证尽量并行推进。
- 默认先交付最小闭环可复核结果，再做增强或重构。
- 信息不全时先做合理假设推进；只有在会引发明显返工、不可逆决策或高风险，或涉及接口、数据模型、破坏性操作时才追问。
- 嵌入式 C 项目优先考虑构建、链接、硬件风险和并发边界。
- 主代理负责拆解、集成、验证和收尾清场。

## 验证原则

### 基本原则
- 验证必须对应实际变更，不做空泛结论。
- 优先使用仓库里已经存在的真实命令。
- 需要证据时，记录命令、结果和残余风险。
- 没有证据时不宣称完成。
- 无法覆盖的验证项必须明确写出未覆盖范围和影响。

### 代码与配置验证
- `AGENTS.md`、`*.md`：检查结构、链接和术语一致性。
- `*.json`、`*.toml`、`*.yaml`：检查语法和层级是否正确。
- 说明类文档：检查是否清楚、是否互相重复、是否存在冲突。

### 构建验证
- `Make` 相关变更：运行真实 `make` 目标。
- `CMake` 相关变更：运行真实 `cmake --build` 或对应脚本。
- 链接、包含路径、宏或生成文件相关变更：必须覆盖到失败点。

### 嵌入式风险验证
- 中断、共享状态、`volatile`、DMA、寄存器和时钟相关改动必须额外检查。
- 板级初始化、启动顺序和外设时序改动必须说明对目标硬件的影响。
- 如果无法在当前环境完成目标机验证，要明确写出残余风险。

### 证据格式
- 命令
- 结果摘要
- 是否通过
- 未覆盖风险

### 结论规则
- 只有证据足够时才能给出通过结论。
- 证据不足时返回 `need-info`。
- 无法继续时返回 `blocked`。

## 确认门槛
仅以下情况暂停确认：
- 破坏性操作：大量删除、重置历史、不可恢复覆盖
- 高成本操作：长时全量测试、重型构建、生产批量发布
- 关键不可逆决策：架构方向、外部契约、数据模型不可逆变更
除以上情况外，默认继续执行，不反复确认。

## 工程质量标准
- 改动必须可运行，禁止只给建议不落地。
- 每轮改动后做关键路径验证。
- 结果必须真实标注：成功/失败/未执行及原因。
- 明确回归风险与影响面。
- 每次交付至少说明变更、验证结果和残余风险。

## 文档落盘规范
- 每次生成文档、计划、纪要、调研、报告、清单时，必须落盘到 `docs` 目录。
- 按类型创建子目录：`docs/plan`、`docs/meeting`、`docs/research`、`docs/spec`、`docs/report`、`docs/decision`、`docs/release`。
- 文件名格式：`<时间戳>_<主题>.md`，时间戳格式 `yyyyMMddHHmmssfff`。
- 若目录不存在，先自动创建再写入。

## 沟通风格
- 全程中文，简洁直接。
- 少背景解释，多给可执行结果。
- 结论直接，避免重复需求和空话。
- 不写空话，不重复需求，不灌水。
- 对未完成项、失败项和未验证项明确标注原因。

## 输出格式
- 结果：一句话总体状态
- Agent进度：按 Agent 标注完成项
- 改动：文件/命令/配置变化
- 验证：执行项与结果
- 管理件：决策记录、任务看板、风险清单
- 下一步：最多3条

## 结束（任务完成时）
- 结束时附上一个简短的 `Stop` 区块：
  - 变更了什么
  - 测试/验证情况（或 `Not run`）
  - 风险/未覆盖项
  - 建议的下一步（如有）
